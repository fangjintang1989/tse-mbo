# Protocol Summary

This is my concise protocol summary based on `protocol_spec.docx`.

## What I Understand Confidently

### 1. Transport and message layout

- The feed is distributed by `UDP/IP` multicast.
- The wire packet is:
  - Ethernet header
  - IP header
  - UDP header
  - tag data
- Tag data is laid out as:
  - `Packet Header`
  - `Tag Length (1)`
  - `Tag Data (1)`
  - ...
  - `Tag Length (n)`
  - `Tag Data (n)`

### 2. Basic encoding rules

- `Char` means ASCII character data, left-aligned.
- `Bn` means unsigned big-endian integer.
- `Bn (Price)` means unsigned big-endian integer price where the last 4 digits are decimal fractions.

### 3. Packet construction rules

- One tag is never split across packets.
- Multiple tags generated at the same time are grouped into one packet.
- One matching or trading-server process can span multiple packets.
- Different issues are not mixed in the same packet.
- Different processes are not mixed in the same packet.

### 4. Packet header

The packet header is `26` bytes and includes:

- `Multicast Group Number`
- `Number of System Reboots`
- `Sequence Number`
- `Issue Code`
- `Update Number`
- `Packet Number`
- `Total Number of Packets`
- `Utility Flag`
- `Message Count`

The fields that matter most for replay are:

- `Sequence Number` for feed ordering
- `Issue Code` for per-issue grouping
- `Update Number` for grouping packets from the same trading-server transaction
- `Utility Flag` for detecting whether more packets from the same matching process still follow

### 5. Tag types in the MBO feed

- `T` Seconds Timestamp
- `O` Trading Status
- `K` Execution Summary
- `A` Add Order
- `E` Order Executed
- `C` Order Executed with Price
- `D` Order Delete
- `R` Reset
- `L` Communication Control
- `II` Issue Information
- `BP` Base Price Information
- `MG` Multicast Group Number Information

### 6. Order-book tags

#### `A` Add Order

Key fields:

- `Order ID`
- `Side`
- `Quantity`
- `Price`
- `Order Condition`
- `Modification Flag`

Meaning:

- creates or updates visible order-book state
- covers both new orders and modification orders
- market orders use the maximum 64-bit price
- order conditions include `Non-conditional`, `On-open`, `On-close`, and `Funari`

#### `D` Order Delete

Key fields:

- `Order ID`
- `Side`
- `Modification Flag`

Meaning:

- removes or invalidates an order
- can correspond to cancel, modification, or expiry

#### `E` Order Executed

Key fields:

- `Order ID`
- `Side`
- `Volume`
- `Match ID`

Meaning:

- execution event for `Zaraba`
- partial executions are possible, so the same order can appear multiple times

#### `C` Order Executed with Price

Key fields:

- `Order ID`
- `Side`
- `Volume`
- `Match ID`
- `Execution Price`
- `Adopted Pricing Method`

Meaning:

- execution event for `Itayose` or quote-display cases
- this is the most auction-relevant execution tag

#### `K` Execution Summary

Key fields:

- `Triggered Side`
- `Total Volume`
- `Total Invalidation`
- `Last Price`
- `Match ID`
- `Best Offer`
- `Best Bid`

Meaning:

- compact summary of the execution result
- useful as validation and as a signal around auction execution behavior

### 7. State and control tags

#### `O` Trading Status

Key fields:

- `Market Status`
- `Status Flag`
- `Short Selling Status`
- `Pricing Method`
- `Book Center Price`

Important explicit values:

- `Market Status`
  - `11` morning order acceptance
  - `12` morning trading hours
  - `19` end of morning session
  - `21` afternoon order acceptance
  - `22` afternoon trading hours
  - `23` afternoon pre-closing
  - `29` end of afternoon session
- `Status Flag`
  - `A0` trading halted, accepting orders
  - `A1` trading halt lifted
  - `B0` Itayose
  - `B1` Itayose ended
  - `C0` trading suspended
  - `C1` trading suspension lifted
  - `D0` trading halted, not accepting orders
- `Pricing Method`
  - `1` Itayose
  - `2` Zaraba
  - `3` sell special quote
  - `4` buy special quote
  - `5` sell sequential trade quote
  - `6` buy sequential trade quote

Meaning:

- `O` tells us the exchange state needed to interpret whether orders and prices should be treated as auction-state or continuous-trading-state information

#### `R` Reset

Meaning:

- reset has start and end markers
- when reset starts, the user must clear all valid previously received `A` tags

#### `L` Communication Control

Key fields:

- `Test Mode Flag`
  - `1` production
  - `2` test
- `Start-End Flag`
  - `1` communication start
  - `2` communication end
  - `0` health check

### 8. Ordering rules

When multiple tags are generated by the same process, the documented priority is:

1. `T`
2. `O`
3. `K`
4. `D`
5. execution and invalidation sequence using `E`, `C`, and `D`
6. `A`

More specific rules:

- on the same price level, execution comes before invalidation
- across multiple price levels, the sequence can be `E, D, E, ...`
- for normal `C` tags in Itayose:
  1. executions at the price where residual orders remain
  2. buy orders from lower price toward market buy
  3. sell orders from higher price toward market sell

### 9. Important appendix rules

#### Pre-closing behavior

- On-close orders become valid after pre-closing starts.
- Funari orders become market orders after pre-closing starts.
- Those transitions are not redistributed as new `A` tags at the transition point.

#### Split packets during auctions

- Large opening-auction Itayose executions can split one matching process across multiple packets.
- `K` may be delivered before all related `C` and `D` packets.
- `Utility Flag` is the main indicator that more packets from the same process still follow.

#### Effective price replacement without new `A` tags

- When initial price or base price changes alter valid upper/lower limits, some existing orders are effectively treated at new limit prices.
- The exchange does not redistribute fresh `A` tags for those adjusted orders.

### 10. What this means for implementation

A correct implementation likely needs:

1. exact feed-order replay
2. per-issue state by `Issue Code`
3. per-order state by `Order ID`
4. reset handling
5. split-packet handling using `Sequence Number`, `Update Number`, and `Utility Flag`
6. auction-aware treatment of `O`, `C`, `K`, and the appendix rules
