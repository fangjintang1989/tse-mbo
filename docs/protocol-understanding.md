# Protocol Understanding

This note is my updated understanding of the protocol based on `protocol_spec.docx`, which is much cleaner than the earlier PDF extraction.

## Source I Used

- `docs/protocol-original.docx`
- `docs/protocol-source.md`
- `docs/jpx-transaction-methods.md`

## High-Level Shape

The protocol is a UDP multicast feed. The distributed packet is described as:

- Ethernet header
- IP header
- UDP header
- tag data

The tag data format is:

- `Packet Header`
- `Tag Length (1)`
- `Tag Data (1)`
- ...
- `Tag Length (n)`
- `Tag Data (n)`

Important transport notes from the docx:

- one tag is never divided across packets
- multiple tags generated at the same time are grouped into one packet
- a single matching process can be split across several packets
- tags for different issues are never mixed in the same packet

## Packet Header

The packet header is `26` bytes long. The fields called out in the docx are:

- `Multicast Group Number`
- `Number of System Reboots`
- `Sequence Number`
- `Issue Code`
- `Update Number`
- `Packet Number`
- `Total Number of Packets`
- `Utility Flag`
- `Message Count`

The fields that look most important for reconstruction are:

- `Issue Code`: per-issue grouping
- `Sequence Number`: feed ordering on a multicast-group basis
- `Update Number`: ties together information generated by one trading-server transaction
- `Utility Flag`: tells us whether more packets from the same matching process follow

## Tag Types

The docx lists these MBO tag types:

- `T`: Seconds Timestamp
- `O`: Trading Status
- `K`: Execution Summary
- `A`: Add Order
- `E`: Order Executed
- `C`: Order Executed with Price
- `D`: Order Delete
- `R`: Reset
- `L`: Communication Control
- `II`: Issue Information
- `BP`: Base Price Information
- `MG`: Multicast Group Number Information

## Tags That Matter Most For The Assignment

### `A` Add Order

Relevant fields listed in the docx:

- `Order ID`
- `Side`
- `Quantity`
- `Price`
- `Order Condition`
- `Modification Flag`

My reading:

- this is the main book-build event
- new orders and modification orders are both represented here
- market orders use a special maximum 64-bit price value
- `Order Condition` includes `Non-conditional`, `On-open`, `On-close`, and `Funari`

### `D` Order Delete

Relevant fields:

- `Order ID`
- `Side`
- `Modification Flag`

My reading:

- this removes visible order-book state
- it covers cancels, priority-changing modifications, and expiry

### `E` Order Executed

Relevant fields:

- `Order ID`
- `Side`
- `Volume`
- `Match ID`

My reading:

- this reduces or removes a live order after Zaraba execution
- one order can appear multiple times because partial fills are allowed

### `C` Order Executed With Price

Relevant fields:

- `Order ID`
- `Side`
- `Volume`
- `Match ID`
- `Execution Price`
- `Adopted Pricing Method`

My reading:

- this is the execution tag that matters for Itayose and quote-display cases
- it is likely important for validating auction behavior and match-price handling

### `K` Execution Summary

Relevant fields:

- `Triggered Side`
- `Total Volume`
- `Total Invalidation`
- `Last Price`
- `Match ID`
- `Best Offer`
- `Best Bid`

My reading:

- `K` is a compact summary of the execution result
- it may be useful both for validation and for identifying auction-related transitions

### `O` Trading Status

Relevant fields:

- `Market Status`
- `Status Flag`
- `Short Selling Status`
- `Pricing Method`
- `Book Center Price`

My reading:

- this tag carries the exchange state needed to understand whether the issue is in order acceptance, trading hours, Itayose, quote display, or related phases
- `Pricing Method` and `Book Center Price` are especially relevant when reasoning about auction-style matching

### `R` Reset

The docx says the user must clear all valid `A` tags when an `R` tag is received.

My reading:

- reset handling is mandatory, not optional
- any replay engine that ignores `R` will produce the wrong book

## Ordering Rules

The docx includes tag priorities for multiple tags generated by the same process:

1. `T`
2. `O`
3. `K`
4. `D`
5. `E` / `C` / `D` ordering around executions and invalidations
6. `A`

It also includes a separate priority rule for normal Itayose `C` tags:

1. orders executed at the price where residual orders remain
2. buy orders from lower price toward market buy
3. sell orders from higher price toward market sell

My reading:

- replay must preserve feed order exactly
- we cannot reorder events by tag type after parsing them

## Auction And Edge-Case Notes That Matter

The appendix in the docx is important.

### Pre-closing and Funari

The docx explicitly says:

- On-close orders become valid on the order book after Pre-closing starts
- Funari orders become market orders after Pre-closing starts
- this transition is not redistributed as fresh `A` tags at the moment Pre-closing starts

My reading:

- a correct user-side book may require rule-based state transitions, not just replay of incoming add/delete/execute tags

### Split packets during opening auctions

The docx explicitly says:

- opening Itayose auctions can generate enough execution data to split one matching process across multiple packets
- `K` can be sent before the remaining packets so the execution summary is delivered without delay
- `Utility Flag` identifies whether more packets from the same matching process still follow

My reading:

- we need to group split packets carefully before drawing conclusions from intermediate states
- `K` arriving early does not mean the matching process is complete

### Replaced effective prices without redistributed `A` tags

The docx explicitly says:

- when the initial price of a newly listed issue is determined, or when a base price changes after order acceptance starts, some existing orders are effectively treated at new limit prices
- `A` tags are not redistributed for those transformed orders

My reading:

- a pure event replay may still be wrong if it ignores exchange-side rules around price-limit changes

## What This Means For The Assignment

For the historical PCAP task, I now expect the final implementation to need:

1. strict packet replay in feed order
2. per-issue state keyed by `Issue Code`
3. per-order state keyed by `Order ID`
4. correct handling of `Utility Flag`, `Sequence Number`, and `Update Number`
5. explicit `R` reset support
6. auction-aware handling of `O`, `C`, `K`, and the appendix rules

## Short Verification Excerpts

These are the short source statements I relied on most heavily:

> The protocol for multicast distribution is UDP/IP.

> The packet distributed consists of an Ethernet header, IP header, UDP header and tag data.

> The tag data consists of a packet header from "3.2.1 Packet Header" and a tag or tags from "3.2.2 Tag Types".

> A tag Add Order 26 Provides information about book registration.

> C tag Order Executed with Price 29 Provides information about registered orders in case of Itayose execution.

> When the user receives this tag, the user must clear all valid A tags.

> In opening auctions or other Itayose auctions, a large amount of execution information is generated.

> If a user constructs an order book or calculates indicative closing prices based on information from the MBO service, it is necessary to consider the above examples on the user-side system.
